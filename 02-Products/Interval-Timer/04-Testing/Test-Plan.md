# 测试计划（Test Plan）- Interval Timer

## 1. 测试概述

### 1.1 测试目标
确保 Interval Timer App V1.0 功能完整、性能稳定、用户体验良好，达到 App Store 上架标准。

### 1.2 测试范围
- **功能测试**：所有 MVP 功能（计时器、预设方案、提醒、统计）
- **UI 测试**：界面显示、交互、动画
- **性能测试**：启动速度、响应速度、资源占用
- **兼容性测试**：iOS 15+、不同设备尺寸
- **用户体验测试**：操作流程、易用性

### 1.3 不包含的测试
- 后台计时功能（V1.0 不支持）
- Apple Watch 版本（V2.0）
- 多语言测试（仅中文）

---

## 2. 测试策略

### 2.1 测试类型

| 测试类型 | 方法 | 工具 | 优先级 |
|---------|------|------|--------|
| 功能测试 | 手动 + 自动化 | Xcode Test | P0 |
| UI 测试 | 手动 | 模拟器 + 真机 | P0 |
| 性能测试 | 自动化 | Xcode Instruments | P1 |
| 兼容性测试 | 手动 | 多个模拟器 | P1 |
| 用户体验测试 | 手动 | 真实用户 | P2 |

### 2.2 测试阶段

#### 阶段 1：单元测试（开发阶段）
- **时间**：开发完成后 1 天
- **负责人**：开发工程师
- **内容**：测试 TimerManager 核心逻辑

#### 阶段 2：集成测试（功能完成后）
- **时间**：单元测试完成后 1 天
- **负责人**：测试工程师
- **内容**：测试各模块集成后的功能

#### 阶段 3：系统测试（集成测试通过后）
- **时间**：2 天
- **负责人**：测试工程师
- **内容**：完整功能测试、UI 测试

#### 阶段 4：验收测试（发布前）
- **时间**：1 天
- **负责人**：产品经理 + 真实用户
- **内容**：验证是否符合 PRD 需求

#### 阶段 5：回归测试（Bug 修复后）
- **时间**：每次 Bug 修复后
- **负责人**：测试工程师
- **内容**：验证 Bug 是否修复，是否引入新问题

## 6. 风险与缓解

### 6.1 风险
| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 测试设备不足 | 无法测试所有机型 | 中 | 使用模拟器 + 借设备 |
| 测试时间不足 | 测试不充分 | 高 | 优先测试 P0 功能 |
| Bug 修复引入新问题 | 质量下降 | 中 | 严格回归测试 |
| 真机测试环境缺失 | 无法测试声音/震动 | 低 | 至少借 1 台真机 |

### 6.2 依赖
- **开发完成**：功能开发必须完成才能测试
- **测试设备**：需要至少 1 台真实 iPhone
- **TestFlight**：需要 Apple Developer 账号
